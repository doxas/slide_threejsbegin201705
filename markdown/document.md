#### やってこない「楽しさ」

結局のところ、私は three.js を一通り書籍にならって使ってみたのですが、今後も継続して利用しようという気持ちにはなれませんでした。

これはひとえに、私自身が既に WebGL のピュアな API の方ばかりを使ってきてしまったために、three.js でラップされてそれらが隠されてしまうと、逆になにが起こっているのかがわからなくなってしまうからなんだと思いました。

---

#### やってこない「楽しさ」

誤解のないように書くと、three.js には、ピュア WebGL で行えるほぼ全てのことを three.js 経由で行うための導線がちゃんと用意されている場合が多いです。

ただ私の場合、その導線（方法）を見つけるためにウェブで調べたりすることよりも、そのまま使う方法が *先にわかってしまっている* ために、どうしても逆に手間に感じてしまうという状態から抜け出せませんでした。

---

#### やってこない「楽しさ」

「こんなの普通に WebGL の API で実装すりゃ一瞬なのに」と思いながらウェブを徘徊するのはつらかったです……

心情としては three.js やそのコントリビュータの方々に対する敬意があるので、余計につらい気持ちになりました。

---

#### ひとそれぞれでいい

でもこれって、逆にこんなふうに考えることもできます。

three.js を利用する際には、ピュア WebGL の API がほとんど見えず、また *それを意識する必要もない* わけです。

それくらい、しっかりと考え抜かれた上で three.js は今も進化し続けています。

---

#### ひとそれぞれでいい

個人的な意見になってしまうのですけれども、3D プログラミングが本当に初めて、という場合は *圧倒的に three.js をおすすめ* します。

ウェブで 3D 表現を行いたいと考える人のほとんどは、コンシューマゲームを開発する生粋のゲームプログラマを目指す人ではなく、あくまでもウェブでなにかを表現したい人のはずです。であれば、できるだけ効率よく、またわかりやすく使えるものを採用すべきです。

---

#### ひとそれぞれでいい

でも、もし 3D プログラミングそのものをより深く知りたいのであれば、three.js はあえて後回しにするのもいいかもしれません。

私の場合はちょっと無理だったのですが、WebGL や OpenGL、あるいは DirectX などを利用した開発への理解があるのなら、three.js を扱う上でも、より柔軟な対応ができるようになるでしょうし、なにより品質の高い作品を生み出せる可能性が高くなると思います。

---

#### ひとそれぞれでいい

three.js は本当によく考えられて実装されています。

その設計を注意深くソースコードレベルで読み解いていくと、さらに 3D プログラミングへの理解を深めるきっかけにもなると思います。

---

#### 自分らしい向き合い方を見つけよう

* three.js はピュアな WebGL の API をほぼ完全に見えなくする
* 同等の機能や作用を three.js の実装として持っている場合がほとんど
* 直感的に記述できコードの記述量も少なくて済む
* ソースコードを追いかけていくとすごく勉強になる点が多い
* バージョンアップが頻繁にあるため最新の状態との差異には気をつけよう
